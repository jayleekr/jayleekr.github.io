# PDR (Product Requirements Document) - Personal Portfolio GitHub Pages

## 1. 문제 정의 (Problem Definition)

### 해결하고자 하는 문제
- **핵심 문제**: 개인의 기술 역량과 프로젝트 경험을 효과적으로 보여줄 수 있는 온라인 포트폴리오의 부재
- **현재 상황**: 
  - 이력서나 LinkedIn 프로필만으로는 개발자의 실제 역량과 프로젝트 세부사항을 충분히 전달하기 어려움
  - 한국어/영어 이중 언어 지원이 필요한 글로벌 기회에 대응하기 어려움
  - 지속적인 업데이트와 유지보수가 체계적으로 이루어지지 않음

## 2. 목표 (Goals/Objectives)

### 비즈니스 목표
- 개인 브랜딩 강화 및 전문성 어필
- 국내외 채용 기회 확대
- 네트워킹 및 협업 기회 창출

### 기술적 목표
- GitHub Pages를 활용한 무료 호스팅
- TDD 기반의 안정적인 코드베이스 구축
- CI/CD를 통한 자동화된 배포 프로세스 구현

## 3. 타겟 사용자 (Target Audience)

### 주요 타겟
1. **채용 담당자/리크루터**
   - 기술 스택과 프로젝트 경험을 빠르게 파악하고자 함
   - 포트폴리오의 완성도를 통해 지원자의 역량 평가

2. **동료 개발자/오픈소스 커뮤니티**
   - 기술 블로그나 프로젝트 문서를 통한 지식 공유
   - 협업 가능성 탐색

3. **잠재적 클라이언트**
   - 프리랜서 프로젝트나 컨설팅 기회 모색

## 4. 핵심 기능 (Key Features)

### 4.1 다국어 지원 (i18n)
- **한국어/영어 전환 기능**
  - 언어 선택 토글 버튼
  - URL 기반 언어 라우팅 (/ko, /en)
  - 브라우저 언어 자동 감지

### 4.2 포트폴리오 섹션
- **About Me**: 자기소개 및 핵심 역량
- **Skills**: 기술 스택 시각화 (레벨 표시)
- **Projects**: 프로젝트 갤러리 (필터링/검색 기능)
- **Blog**: 기술 블로그 (Markdown 기반)
- **Contact**: 연락처 및 소셜 링크

### 4.3 프로젝트 상세 페이지
- 프로젝트 개요 및 배경
- 사용 기술 스택
- 주요 기능 및 구현 내용
- 데모 링크 및 GitHub 저장소
- 스크린샷/GIF 데모

### 4.4 블로그 기능
- Markdown 기반 포스트 작성
- 카테고리 및 태그 분류
- 검색 기능
- RSS 피드 제공

## 5. 비기능 요구사항 (Non-functional Requirements)

### 5.1 성능
- Lighthouse 점수 90점 이상
- 첫 페이지 로드 시간 3초 이내
- 이미지 최적화 (WebP, lazy loading)

### 5.2 접근성
- WCAG 2.1 AA 준수
- 키보드 네비게이션 완벽 지원
- 스크린 리더 호환성

### 5.3 SEO
- 메타 태그 최적화
- Open Graph 태그
- 구조화된 데이터 (JSON-LD)
- Sitemap 자동 생성

### 5.4 디자인
- 반응형 디자인 (모바일 우선)
- 다크/라이트 테마 전환
- 일관된 디자인 시스템

## 6. 성공 지표 (Success Metrics)

### 정량적 지표
- 월간 방문자 수
- 평균 체류 시간
- 페이지뷰 수
- Contact 폼 제출 횟수
- GitHub 프로필 클릭률

### 정성적 지표
- 포트폴리오를 통한 면접 기회 증가
- 네트워킹/협업 제안 수
- 블로그 포스트 공유/인용 횟수

## 7. 기술 스택 및 개발 방법론

### 7.1 프론트엔드
- **프레임워크**: Next.js 14+ (App Router)
- **스타일링**: Tailwind CSS + CSS Modules
- **상태관리**: Zustand
- **다국어**: next-i18next

### 7.2 테스팅 (TDD)
- **단위 테스트**: Jest + React Testing Library
- **통합 테스트**: Cypress
- **시각적 회귀 테스트**: Percy
- **테스트 커버리지**: 80% 이상

### 7.3 개발 도구
- **번들러**: Webpack (Next.js 내장)
- **린터**: ESLint + Prettier
- **타입 체크**: TypeScript
- **커밋 규칙**: Conventional Commits + Husky

### 7.4 CI/CD
- **GitHub Actions**:
  - PR 시 자동 테스트 실행
  - 메인 브랜치 푸시 시 자동 배포
  - 일일 빌드 상태 체크

## 8. 가정 및 제약사항

### 가정
- GitHub Pages의 정적 사이트 제한 사항 수용
- 방문자는 주로 데스크톱/모바일 최신 브라우저 사용
- 콘텐츠는 주기적으로 업데이트됨

### 제약사항
- **호스팅**: GitHub Pages 무료 플랜 (1GB 저장소, 월 100GB 대역폭)
- **백엔드**: 서버리스 함수 또는 외부 API만 사용
- **데이터베이스**: 정적 JSON 파일 또는 Headless CMS
- **도메인**: jayleekr.github.io (커스텀 도메인 추후 고려)

## 10. 테스트 전략

### TDD 프로세스
1. **Red**: 실패하는 테스트 작성
2. **Green**: 테스트를 통과하는 최소한의 코드 작성
3. **Refactor**: 코드 개선 및 최적화

### 테스트 종류
- **컴포넌트 테스트**: 각 UI 컴포넌트의 렌더링 및 상호작용
- **다국어 테스트**: 언어 전환 시 올바른 텍스트 표시
- **라우팅 테스트**: 페이지 간 이동 및 URL 처리
- **접근성 테스트**: 키보드 네비게이션, ARIA 속성
- **성능 테스트**: 로딩 시간, 번들 크기

## 11. Git 전략

### 브랜치 전략
- `main`: 프로덕션 배포 브랜치
- `develop`: 개발 통합 브랜치
- `feature/*`: 기능 개발 브랜치
- `fix/*`: 버그 수정 브랜치
- `docs/*`: 문서 업데이트 브랜치

### 커밋 메시지 규칙
```
<type>(<scope>): <subject>

<body>

<footer>
```

Types: feat, fix, docs, style, refactor, test, chore
